# AI时代的知识库管理：从Wiki到工程化学习笔记

为什么现在要重新思考知识库 过去十几年，软件团队的知识库基本是一个模式：Confluence、Wiki、Notion，写给人看，靠人维护，靠人检索。这套范式正在快速失效——不是因为工具不好，而是因为读者变了，写者也变了。新增的读者是AI。它消费知识的方式和人完全不同：人是稀疏、叙事性、按需读的；AI是稠密、穷尽、结构化地读。一份对人友好的文档，对AI可能毫无价值。新增的写者也是AI。它能从代码中抽取知识、自动生成总结、起草文档初稿。这意味着知识库正在从”人单向写、人单向读“的产物，变成”人机协同生产、人机共同消费“的工程系统。

# 知识库给谁看

人+AI双一等公民读者；AI同时是写者，人是AI产出的reviewer

如果设计成”主要给AI看“：那就会存在两个问题：

  1. 失去校验能力。因为AI不能锚定ground truth，会产生幻觉。知识只有AI在写、AI在读，错误就会自我增殖。必须有人在回路。
  2. 忽略AI同时是写者。工作流的重心从”写“转移到了”审“——产能的瓶颈从内容生产变成了内容验证。

因此知识库设计取向应该是：机器可解析结构+人类可读，两类读者同时服务，不能二选一。任何”全自动AI维护“方案注定失败

# 知识库应该管什么

只管不能从代码现场推导的知识

凡是从代码推导出的，不要存。一旦存下，就和代码drift；知识就成了陷阱而不是资产。真正进知识库的应该只有五类：

类目说明为什么必须显式存
意图与决策为什么选X不选Y代码说what，不说why
负向知识试过什么、为什么放弃最稀缺，只在踩过坑的人脑子里
边界与所有权谁调谁、出事找谁散落在组织结构图外
运维与操作怎么跑、怎么部、怎么debug半条命在某人terminal history里
领域语义与不变量业务术语、边界条件代码编码了但不解释

知识库应该比直觉小的多。

# 知识库怎么管

当软件项目运营，以冷启动可验证为唯一验收标准

每一份知识都要走完一个完整的生命周期，每个生命周期阶段都要有对应的机制兜底。

生命周期阶段典型失败模式对应机制
写入错的或低质内容混入PR工作流+信任分级
存放不同类型知识互相污染三层结构+Harness分级
使用AI/人找不到需要的内容结构性可发现性
验证内容悄悄偏离真实Cold-start test
演化越积越多直到维护成本崩溃衰减监控+季度剪裁

# 写入：PR工作流+信任分级

AI生成的内容必须人工review后入库，不允许agent自动commit。

入库后，必须分级。必须遵守的，团队约定的，参考信息，AI起草未人工验证的。没有分级就会存在隐患。

# 存放：三层结构+Harness 分离

要防的是”信号被噪音淹没“

热核心层（进每次上下文），体量要小，指针式，大量交叉引用而非展开。内容比如项目骨架、关键约定、入口指引。可以放到AGENTS.md
温层指南（按需调入），由热核心层显式指向。内容比如领域语义、边界。可以放到docs/或者 docs/guide
冷归档层（可检索但不常读）内容比如负向知识、事故复盘。放docs/history/、docs/decisions/
Harness层（工具，不是文档）跟文档分开

# 使用：结构性可发现性

防止存了找不到

不要把向量检索当主力。Agent会自己grep、glob、递归看目录。结构性可发现性比向量质量更重要：

  • 命名规范统一： {module}-{type}-{topic}.md 可预测
  • 目录布局可预测：agent看根目录能猜到去哪找
  • 交叉引用准确：相对路径，CI校验链接有效性
  • 索引文件：每个docs/ 子目录有 README.md清单

向量检索可以当补充手段

# 验证

要防自我感觉良好的腐烂

模拟最坏情况，模拟一个完全没有先验上下文的AI agent，只给它知识库，看能不能完成任务

# 演化：衰减监控+季度裁剪

要防熵增导致的维护破产

  1. 衰减信号监控（脚本）
  2. 季度裁剪

# 避坑清单

  1. 集中式”知识中台“ —————— 没人维护
  2. 让AI自动写不review ———— 迅速低质
  3. 追求完整覆盖 ———— 应该追求关键路径正确
  4. 把Harness和知识混在一个目录 ———— 工具和文档要分开
  5. 用文档替代可执行物 ———— 能写成script的不写成md
  6. 忽略负向知识 ———— 最稀缺、最被低估的资产
  7. 追求文档数量指标 ———— 数量是反指标，质量与可验证性才是

# 业务知识

领域语言层（Domain Language）：订单、客户、账户。这些高频稳定跨需求复用，体量小。
业务规则层（Business Rules）：夜间订单走异步通道、金额超过xx风控。中频半稳定。体量中等按主题切分。
历史决策层（Historical Decisions）：为什么X字段允许为空。低频。体量大但分散。
当下需求层（Current Context）： 这次做啥。临时高强度相关。体量看需求。
